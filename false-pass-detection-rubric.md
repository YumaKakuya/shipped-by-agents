# False PASS Detection Rubric

A quick-reference rubric for identifying false passes in AI agent-generated code.

A **False PASS** is when all tests pass, CI is green, and the code still doesn't work. It's the most dangerous failure mode in AI-assisted development because it looks like success.

## The 5-Point Check

Run this against any AI agent-generated PR before merging.

### 1. Assertion Relevance

> Do the tests assert what actually matters, or just what's easy to assert?

| Signal | Status |
|--------|--------|
| Tests check return values against hardcoded expected outputs | Suspect |
| Tests verify behavior under realistic input conditions | OK |
| Tests only check that "no error was thrown" | False PASS risk |
| Tests validate integration points with adjacent systems | OK |

### 2. Test-to-Requirement Traceability

> Can you trace each test back to an actual requirement?

| Signal | Status |
|--------|--------|
| Agent wrote tests that match its own implementation (circular) | False PASS risk |
| Tests were derived from a spec or requirement doc | OK |
| Tests exist but no one can explain what user behavior they validate | False PASS risk |
| Test names describe user-facing outcomes | OK |

### 3. Input Realism

> Are test inputs representative of real-world usage?

| Signal | Status |
|--------|--------|
| All test inputs are simple, happy-path values | Suspect |
| Edge cases, empty states, and boundary values are covered | OK |
| Test data was generated by the same agent that wrote the code | False PASS risk |
| Inputs include known problematic patterns from production | OK |

### 4. Mutation Sensitivity

> If you break the code, do the tests actually fail?

| Signal | Status |
|--------|--------|
| Commenting out core logic still leaves tests green | False PASS confirmed |
| Changing a key conditional flips at least one test to red | OK |
| Tests mock so heavily that the real code path is never executed | False PASS risk |
| Integration tests hit actual dependencies (DB, API, filesystem) | OK |

### 5. Scope Completeness

> Does the agent's output cover what was actually asked for?

| Signal | Status |
|--------|--------|
| PR implements 3 of 5 requested features but all 3 pass tests | Incomplete, not False PASS — but often misreported as "done" |
| Agent added features not requested, with tests for those instead | Scope drift + False PASS risk |
| Every acceptance criterion has at least one corresponding test | OK |
| "All tests pass" is the only evidence of completion | False PASS risk |

## Quick Decision

After running the 5-point check:

- **0 flags**: Proceed to review with normal scrutiny.
- **1-2 Suspect flags**: Investigate those areas. Likely fixable.
- **Any "False PASS risk" flag**: Do not merge. Require the agent (or a human) to rewrite tests for flagged areas.
- **Any "False PASS confirmed"**: Stop. The test suite is providing false confidence. Rewrite tests from requirements before re-evaluating the code.
